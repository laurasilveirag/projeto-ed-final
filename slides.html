O Ponto de Partida

De onde vêm os dados?

Ambiente de Origem

Banco de dados relacional PostgreSQL 16 simulando um serviço de streaming de música. 12 tabelas interligadas, dados distribuídos nos últimos 3 anos, gerados com a biblioteca Faker.

Volume de Dados

70.000 reproduções
10.500 pagamentos
10.000 músicas
3.000 usuários
500 artistas

O desafio: transformar esse banco relacional transacional em um modelo analítico pronto para geração de KPIs e dashboards.

O que acontece aqui

O PySpark lê as tabelas do Postgres via JDBC e salva cada uma como arquivo CSV com header no bucket landing do MinIO. Dados brutos, sem nenhuma transformação.

Por que CSV?

O professor especificou formato bruto original — CSV para dados SQL. A Landing é a zona de quarentena: o dado chega exatamente como saiu da origem.

8 dimensões 3 tabelas fato MinIO bucket: landing/

Camada 2 — Promoção

Bronze — A Cópia Fiel

Landing
CSV
──Delta──▶
Bronze
Delta Lake
Schema
Os CSVs são lidos com schemas explícitos definidos a partir do DDL de origem. Cada coluna recebe o tipo correto — inteiros, timestamps, decimais, booleans.
Delta
O dado é persistido em formato Delta Lake — Parquet com transaction log. Isso garante operações ACID, time travel e schema enforcement.
Partição
Particionado por ingestao_date — a data em que o dado entrou no lake. Rastreabilidade total de quando cada lote foi ingerido.

Limpeza Geral

· Deduplicação por id em todas as tabelas
· Trim de espaços em colunas de texto
· Padronização de status com lower()
· Dimensões sem partição (volume pequeno)

Regras de Negócio — Fatos

· Plays com ms_tocados < 30.000 ms são descartados
· completou é recalculado: ms_tocados ≥ 90% da duração
· Fatos particionados por ano_mes do evento
· 7 testes pytest validam as regras

O Metabase não lê Delta diretamente — a Gold é espelhada no Postgres (schema gold) via JDBC ao final de cada run do Airflow.

Orquestração

Apache Airflow — DAG Completa

criar_buckets
  ├── landing_dimensoes → bronze_dimensoes → silver_dimensoes → gold_dimensoes ──┐
  └── landing_fatos → bronze_fatos → silver_fatos → gold_fatos ──┤
                                                                                        gold_postgres

Agendamento

Execução diária (@hour). A DAG roda automaticamente e processa apenas os dados novos.

Carga Incremental

Watermark por created_at — cada run lê só os registros inseridos após o último processamento.

10 a 20 min

Tempo de execução completo do pipeline, rodando PySpark in-process no container do Airflow.
