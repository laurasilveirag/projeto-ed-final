# ADR-0004 — Spark via PythonOperator no Airflow

**Status:** Aceito  
**Data:** 2026-06  
**Closes:** #41

## Contexto

O Airflow precisa executar transformações PySpark. Existem três abordagens:

1. **SparkSubmitOperator** — submete jobs para um cluster Spark externo
2. **DockerOperator** — executa Spark em containers separados
3. **PythonOperator** — roda PySpark in-process no worker do Airflow

## Decisão

Usar **PythonOperator** com PySpark in-process no container do Airflow.

## Justificativa

- Volume de dados é pequeno (~100k linhas) — não precisa de cluster
- Simplifica a infraestrutura: uma única imagem Docker (Airflow + PySpark + Java 17)
- Sem overhead de comunicação com cluster externo
- Mais fácil de debugar (logs direto no Airflow)

## Implementação

```dockerfile
# docker/airflow/Dockerfile
FROM apache/airflow:2.9.2-python3.11
USER root
RUN apt-get install -y openjdk-17-jdk
USER airflow
RUN pip install pyspark==3.5.1 delta-spark==3.2.0
```

```python
# Na DAG
from airflow.operators.python import PythonOperator

task = PythonOperator(
    task_id="silver_fatos",
    python_callable=silver_fatos,  # função Python que usa SparkSession
)
```

## Consequências

- **Positivas:** infraestrutura simples, um único container, fácil manutenção
- **Negativas:** não escala para volumes grandes; Spark ocupa memória do worker
- **Limitação:** SparkSession precisa de `spark.hadoop.fs.defaultFS=file:///` para não conflitar com a porta 9000 do MinIO
